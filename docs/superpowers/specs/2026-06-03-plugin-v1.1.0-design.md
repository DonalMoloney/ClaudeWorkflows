# Plugin v1.1.0 Design Spec

**Date:** 2026-06-03
**Scope:** parallel-workflows Claude Code plugin at `plugin/`
**Goal:** Harden the plugin as a production reference implementation others clone and extend.

---

## What changes and why

### Approach A — Quality pass (6 edits)

**`commands/orchestrate.md`**
Remove the duplicated adversarial-verify and loop-until-dry code blocks. Replace with a single cross-reference line pointing readers to the `parallel-patterns` skill for authoritative templates. Eliminates drift risk between two copies of the same patterns.

**`commands/parallel-sweep.md`**
Add `version: 1.0.0` to frontmatter. Consistency with all other components.

**`skills/dynamic-workflows/SKILL.md`**
Add a **Troubleshooting** section with five common mistakes and their fixes:
1. Using `Date.now()` / `Math.random()` / `new Date()` — breaks resume
2. TypeScript syntax (type annotations, interfaces, generics) — runtime rejects it
3. Missing `.filter(Boolean)` after `parallel()` — null results cause downstream crashes
4. Using `parallel()` as a default — should only be used when a stage genuinely needs all prior results
5. Schema field mismatch — model retries internally; don't add manual retry logic on top

Also expand the Constraints table to note tool scoping: sub-agents inherit the session's tool allowlist; shell commands/web fetches/MCP tools not on the allowlist can still prompt mid-run.

**`agents/sub-agent-specialist.md`**
Add two new sections:
- **Tool inheritance** — sub-agents inherit the session allowlist; use `--allowedTools` in headless mode to restrict
- **Agent failure behaviour** — a crashed or out-of-tokens agent returns `null`; always `.filter(Boolean)` before iterating results

**`.claude-plugin/plugin.json`**
Bump version to `1.1.0`.

**`README.md`** (new file at `plugin/README.md`)
~120-word orientation document: what the plugin is, its four components (commands, agents, skills, plugin.json), how they relate, and the typical usage flow (`/orchestrate` → write script → `ultracode` trigger → `/workflows`).

### Approach B — New skill: error handling

**`plugin/skills/error-handling/SKILL.md`**

Covers the production failure modes the other files don't address:

- **Agent `null` returns** — what causes them (timeout, token limit, tool denial, schema mismatch exhaustion) and how to handle them (`.filter(Boolean)`, log dropped items, never silently continue)
- **Defensive schema design** — required fields, no nested objects as field values, enum for bounded strings, avoid optional fields that Claude will omit and break downstream code
- **`.filter(Boolean)` placement** — after every `parallel()` call; inside `pipeline()` stages that receive arrays; before any `.map()` / `.flatMap()` on agent results
- **`try/catch` in pipeline stages** — drop a failing item without aborting the whole run; log what was dropped so coverage is visible
- **Schema retry behaviour** — the model retries on mismatch automatically; adding manual retry logic creates double-retry amplification

### Approach C — New command: workflow review

**`plugin/commands/workflow-review.md`**

`/workflow-review <path>` reads the script at `<path>` and delegates to the `workflow-architect` agent to critique it across six dimensions:

| Dimension | What's checked |
|---|---|
| Meta declaration | `export const meta` is a pure literal; `phases` array matches `phase()` calls |
| Pipeline vs parallel | `parallel()` used only where a barrier is genuinely needed |
| Label/phase assignment | Every `agent()` has a `label`; agents inside `pipeline()`/`parallel()` have `phase` set |
| Schema quality | Schemas are tight (required fields, enum for bounded values, no nested objects as values) |
| No-random/no-Date | No `Date.now()`, `Math.random()`, or `new Date()` |
| Agent count realism | No unbounded loops without `budget.remaining()` guard or dry-streak exit condition |

Output: structured findings list with severity (`error` / `warning` / `suggestion`) and a one-line fix for each.
